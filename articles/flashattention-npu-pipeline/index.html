如何在NPU设计中加速 FlashAttention 计算

聚焦 Online Softmax 的 loop-carried dependency，用 lookahead 展开把标量递推链路做成高吞吐全流水结构。

NPU设计NPUPipelineLookahead2026-06-20

上一篇主要讲了FlashAttention 的大致硬件计算流程。

这篇换一个视角：如果要在推理芯片/NPU 上实现 FlashAttention，怎样把它做成全流水、高吞吐结构？

矩阵乘、load/store、rowmax、exp、rowsum等这些步骤本身都比较容易做流水。真正容易卡住的地方，反而是 Online Softmax 里的部分和更新：

\[\ell_n=a_n\ell_{n-1}+r_n\]

其中：

\[a_n=\exp(m_{n-1}-m_n)\]

这个式子看起来很简单，但它有一个麻烦的 loop-carried dependency：当前 \(\ell_n\) 依赖上一拍的 \(\ell_{n-1}\)，这一步需要算浮点乘 浮点加，很显然一拍算不完。

如果这条标量链路不能做到一拍一个结果，那么前面再大的矩阵乘吞吐也会在这里被限流。

1. FlashAttention 的硬件流水视角

以单头 attention 为例：

  • \(Q \in \mathbb{R}^{N\times d}\)；
  • \(K,V \in \mathbb{R}^{N\times d}\)；
  • \(Q\) 按行切成 \(B_r\times d\) 的小块；
  • \(K,V\) 按行切成 \(B_c\times d\) 的小块。

对于一个 query block：

\[Q_i\in \mathbb{R}^{B_r\times d}\]

遍历所有 key/value block：

\[K_j,V_j\in \mathbb{R}^{B_c\times d}\]

当前 score block 为： \(S_i^{(j)}=Q_iK_j^T\)

FlashAttention 分块计算示意图

从芯片实现角度，可以抽象为若干步骤：

Tensor--矩阵运算单元；Vect.--向量运算单元；Acvt.--激活函数运算单元；Scalar--标量运算单元；

  1. 搬运小块矩阵到片上 SRAM；
  2. Tesnsor计算 \(S_i^{(j)}=Q_iK_j^T\)；
  3. Vect对 \(S_i^{(j)}\) 做 rowmax以及 \(S_i^{(j)}-m_i\) ；
  4. Acvt计算 \({\widetilde{\mathbf{P}}}_i^{\left(j\right)}=\exp{\left(\mathbf{S}_i^{\left(j\right)}-m_i\right)}\) ，同时Scalar计算 \(a_i=\exp{\left(m_i^{\mathrm{old}}-m_i\right)}\) ；
  5. Vect对上步输出的exp值做rowsum 得到当前 block 的 denominator 增量 \(r_i\) ；
  6. Scalar更新 \(\ell_i=a_i\ell_i+r_i\) ；
  7. Tensor算 \({\widetilde{\mathbf{P}}}_i^{\left(j\right)}\mathbf{V}_j\)；
  8. Vect更新 \({\widetilde{\mathbf{O}}}_i={a_i}{\widetilde{\mathbf{O}}}_{i}+{\widetilde{\mathbf{P}}}_i^{\left(j\right)}\mathbf{V}_j\)
  9. 遍历完所有j块后，scalar算 \(\ell_i^{-1}\) ；
  10. 最后tensor做\(\mathbf{O}_i=\mathrm{diag}\left(\ell_i\right)^{-1}{\widetilde{\mathbf{O}}}_i\) ，或vect做 \(\mathbf{O}_i=\ell_i^{-1}{\widetilde{\mathbf{O}}}_i\) 。

前面几级都比较常规，均可以在特定engine内实现流水化计算，但是到了第 6 步，就有点问题了。第6步等号左侧的 \(li\) 是新的，等号右侧的 \(li\) 是上拍的。单独拎出来该公式如下：

\(\ell_n=a_n\ell_{n-1}+r_n\)用 \(n\) 表示当前拍， \(n-1\) 为上拍，其中 \(a_n=\exp(m_{n-1}-m_n)\) 。该系数的运算\(exp{\left(m_i^{\mathrm{old}}-m_i\right)}\) 在步骤3之后就可由scalar进行 。到了第6步，还需要一个乘法 一个加法才能完成。即有了 \(\ell_{n-1}\) 后，还需要对他进行乘法 + 加法操作。

这两步的浮点运算，在FP32~FP20格式范围内，一拍很难算完，假定这里2拍可以搞定。这块没法直接做成打拍的多级流水，因为每2拍才输出一个新的 \(\ell_n\) ，这就是典型的loop-carried dependency，即本次循环迭代 \(\ell_n\) 的计算，必须等待上一轮迭代 \(\ell_{n-1}\) 的输出结果，而每轮 \(\ell_n\) 的计算需要两拍时，整个流水线的吞吐率就被砍半了。

下面进行更直观的拆解步骤。

2. 步骤6流水降速拆解

假设每拍都能产生新的 \(a_n\) 和 \(r_n\)，理想情况当然是每拍也吐出一个新的 \(\ell_n\)。

但该处更新至少包含：

  1. 乘法：\(a_n\ell_{n-1}\)；
  2. 加法：\(a_n\ell_{n-1}+r_n\)。

如果乘加路径需要两拍，就会变成：

第 t 拍：拿到 l_{n-1}
第 t+1 拍：完成乘法
第 t+2 拍：完成加法，得到 l_n

下一项 \(\ell_{n+1}\) 又必须等 \(\ell_n\) 出来。

这样一来，整条流水可能两拍才能接收一个新的 micro-block进行计算，initiation interval 从 1 变成 2。

对 NPU/推理芯片来说，这很致命：QK、exp、rowsum、PV 都能持续流，最后被一个标量递推卡住吞吐。

3. 用 Lookahead 展开打破一拍依赖

既然 \(\ell_n\) 依赖 \(\ell_{n-1}\) ,而 \(\ell_{n-1}\) 又依赖 \(\ell_{n-2}\) ,那么\(\ell_n\) 就必定间接依赖 \(\ell_{n-2}\) 。如果将 \(\ell_n\) 转换为 \(\ell_{n-2}\) 的表达式，即便每两拍出一个新的\(\ell_{n+2}\)，那么实际上仍等价于每拍出一个新的\(\ell_{n+1}\)：

先写原始递推：

\[\ell_n=a_n\ell_{n-1}+r_n\]

上一项为：

\[\ell_{n-1}=a_{n-1}\ell_{n-2}+r_{n-1}\]

代入：

\[\ell_n = a_n(a_{n-1}\ell_{n-2}+r_{n-1})+r_n\]

展开：

\[\ell_n = a_na_{n-1}\ell_{n-2} + a_nr_{n-1} + r_n\]

再把 \(a_n\) 写回 exp 形式：

\[a_n=\exp(m_{n-1}-m_n)\]
\[a_{n-1}=\exp(m_{n-2}-m_{n-1})\]

所以：

\[a_na_{n-1} = \exp(m_{n-2}-m_n)\]

最终得到：

\[\ell_n = \exp(m_{n-2}-m_n)\ell_{n-2} + \exp(m_{n-1}-m_n)r_{n-1} + r_n\]

写成硬件更直观的形式：

\[\ell_n = scale_{acc}\cdot \ell_{n-2} + scale_{prev}\cdot r_{n-1} + r_n\]

其中：

\[scale_{acc}=\exp(m_{n-2}-m_n)\]
\[scale_{prev}=\exp(m_{n-1}-m_n)\]

可以看到从 \(\ell_n \leftarrow \ell_{n-1}\) 转变为了 \(\ell_n \leftarrow \ell_{n-2}\) 。

要计算出最终的 \(\ell_n\) ，下述两套计算链路均可以实现：

偶数链：l_0 -> l_2 -> l_4 -> l_6 -> ...
奇数链：l_1 -> l_3 -> l_5 -> l_7 -> ...

所以我们仅需在硬件中做出偶数链或奇数链。以偶数链为例，如果最终的 \(n\) 是偶数，直接就能算出；如果最终的 \(n\) 是奇数，那么在最后一步稍微处理一下，重新按照 \(\ell_n=a_n\ell_{n-1}+r_n\) 递推公式计算即可。

这不是近似算法，也没有改变 softmax 的数学意义。只是把同一个递推式展开了一步，把依赖距离从 1 拉到 2。

4. 硬件里需要维护什么？

根据：

\[\ell_n = \exp(m_{n-2}-m_n)\ell_{n-2} + \exp(m_{n-1}-m_n)r_{n-1} + r_n\]

硬件里需要维护：

  • \(\ell_{n-2}\)：两拍前的部分和；
  • \(r_{n-1}\)：上一拍的 rowsum；
  • \(m_{n-2}\)：两拍前的最大值；
  • \(m_{n-1}\)：上一拍的最大值；
  • 当前拍的 \(m_n\) 和 \(r_n\)。

当前拍 rowmax 得到 \(m_n\) 后，采用流水依次计算两个系数：

\[scale_{prev}=\exp(m_{n-1}-m_n)\]
\[scale_{acc}=\exp(m_{n-2}-m_n)\]

然后：

\[tmp=scale_{prev}\cdot r_{n-1}+r_n\]
\[\ell_n=scale_{acc}\cdot \ell_{n-2}+tmp\]
第 t 拍  ：算scale_{prev}
第 t+1 拍：算scale_{acc}
...假设exp流水线为6拍...
第 t+ 6拍：scale_{prev}出结果，与r_{n-1}相乘
第 t+ 7拍：scale_{prev} * r_{n-1}出结果，这个结果 + r_{n}算tmp;scale_{acc}出结果，与l_{n-2}相乘
第 t+ 8拍：tmp出结果；scale_{acc} * l_{n-2}出结果；二者再相加
第 t+ 8拍：l_{n}出结果

5. 延伸-反馈延迟深度超过两拍

两步展开适合反馈延迟为两拍的情况。

如果反馈路径需要 \(L\) 拍，可以继续做 \(L\) 步 lookahead。

原始递推：

\[\ell_n=a_n\ell_{n-1}+r_n\]

本质上是一个 affine transform：

\[T_n(x)=a_nx+r_n\]

多个 step 可以组合：

\[T_n(T_{n-1}(x)) = (a_na_{n-1})x+(a_nr_{n-1}+r_n)\]

更一般地，连续 \(L\) 个 step 可以写成：

\[\ell_n=A_n\ell_{n-L}+B_n\]

其中 \(A_n\) 是一串 scale 的乘积，\(B_n\) 是一串经过 scale 修正后的 rowsum 累加。

从更一般的角度看，这就是 affine recurrence 的 lookahead，也可以理解为一种 prefix/scan 化的实现思路。

6. O 的更新也可以做类似展开

denominator 是：

\[\ell_n=a_n\ell_{n-1}+r_n\]

output numerator 是：

\[O_n=a_nO_{n-1}+U_n\]

其中：

\[U_n=P_nV_n\]

同样展开一步：

\[O_n = a_na_{n-1}O_{n-2} + a_nU_{n-1} + U_n\]

也就是：

\[O_n = \exp(m_{n-2}-m_n)O_{n-2} + \exp(m_{n-1}-m_n)U_{n-1} + U_n\]

不过 \(O_n\) 不是标量，通常会由更宽的 vect/tensor pipeline 承担；\(\ell_n\) 是每行一个标量，反而更容易成为控制和归一化链路上的瓶颈。

所以如果要突出“最难流水起来的地方”，\(\ell_n\) 是一个非常好的切入点。

7. 小结

FlashAttention 在推理芯片上要跑满，不只是 QK 和 PV 要快。

Online Softmax 里的递推：

\[\ell_n=a_n\ell_{n-1}+r_n\]

会形成loop-carried dependency。如果这里是 \(n\) 拍完成，如果不处理，整条FlashAttention流水线的吞吐率将会降为全流水的 \(1/n\) 。

通过 lookahead 展开：

\[\ell_n = a_na_{n-1}\ell_{n-2} + a_nr_{n-1} + r_n\]

可以把依赖从 \(\ell_{n-1}\) 拉到 \(\ell_{n-2}\)，再用奇数链或偶数链进行计算，把两拍反馈链路做成等价于每拍均输出的高吞吐全流水结构。

全文可总结为：

所以说，不只是 QK 和 PV 要快，整个Online Softmax都要快， 尤其 \(\ell\) 和 \(O\) 的递推链路。可以对整个FlashAttention做全流水化处理；算法上的 Online Softmax 解决“分块怎么算对”，而 NPU 上的全流水实现还要解决“递推链路怎么不断流”；\(\ell_n=a_n\ell_{n-1}+r_n\) 的 lookahead 展开，就是打破这个短反馈瓶颈的关键。
