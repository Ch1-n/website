SoC 芯片低功耗设计全流程

从 Power 与 Energy 的目标区分出发，梳理动态功耗、静态功耗、系统架构、模块 RTL、Memory、物理实现和 Golden UPF Flow 的低功耗设计路径。

低功耗设计SoC低功耗UPF2026-06-27

前段时间做了一些简单的低功耗设计，总结了点经验，这里做一个简单的记录。

1. 功耗概述

本小节主要讲述一下优化目标、功耗组成、主要占比等内容。

做低功耗设计之前，第一步通常不是敲定低功耗策略，而是结合应用场景确认优化侧重点。

功耗、能量、平均功耗、峰值功耗、动态功耗、漏电功耗等，当然并不是只针对其中一项进行优化，而且多路齐下，分个轻重缓急以最大化收益。

1.1 Power or Energy？

我们通常所说的低功耗大多指降低芯片功率，但这只是优化目标之一。功率在时间维度上的累计称为能量，如何降低芯片完成一项任务所消耗的总能量，也同等甚至更加重要。

Power 是单位时间消耗的能量，Energy 是总能量：

Energy = Power * worktime

很多设计讨论里容易把“低功耗”和“低能耗”混在一起。二者相关，但不完全等价。

如果一个方案功率更低，但运行时间被拉长很多，总能量未必更低。反过来，一个方案峰值功率更高，但能更快完成任务并进入休眠，总能量可能更优。

因此：

  • 对于电池驱动、移动终端、IoT 等场景，更关心 Energy，即一次任务需消耗多少电。
  • 对于持续供电、需trade off 延迟、漏电、性能、封装散热约束等场景，更关心 Power。

举个很简单的例子，对于手机soc，我们的关注侧重点是Energy。但对于桌面级GPU，我们更关心其Power。

1.2 Average or Peak

降低平均功耗，本质是减少系统整体能耗。

而峰值功耗过高会引发多重工程风险：

  • 散热压力增大，性能释放不稳定。
  • 后端APR阶段电源网络IR不收敛。
  • 容易出现可靠性问题，降低使用寿命等。

1.3 Dynamic or Static

  • 对持续高负载运行的模块，动态功耗是大头。
  • 对长期 standby 的模块，静态功耗更值得关注。

1.4 SoC 功耗组成

数字芯片功耗通常可以粗略分成两类：动态功耗和静态功耗。

1.4.1 动态功耗：翻转产生的功耗

动态功耗主要来自节点翻转、电容充放电、短路电流等。常见形式可以理解为：

Pdyn ≈ (Ceff * Vdd^2 * f) + (t * Vdd * Ipeak * f)

这里不对公式做详细说明，如有疑问可查阅其他帖子。

我们关注的重点是，动态功耗与电压平方、频率及电容成相关。

所有的降功耗路子追根到底都是从公式上下手，以上式为例：

  • 降电压：multi-voltage、DVS（Dynamic Voltage Scaling）、DVFS（Dynamic Voltage and Frequency Scaling）、AVFS（Adaptive Voltage and Frequency Scaling）。
  • 降频率：DFS（Dynamic Frequency Scaling）、DVFS、AVFS。
  • 降翻转：clock gating、data gating、operand isolation、减少无效数据传播等。
  • 降电容：长沟道管子 HVT器件 LP工艺。

1.4.2 静态功耗：漏电产生的功耗

静态功耗主要来自器件漏电，包括 sub-threshold leakage、gate leakage、junction leakage 等。随着工艺演进，漏电越来越不能忽视。

Pleak ≈ （Isub + Igate + Ijunc）* Vdd

常见降低静态功耗的手段包括：

  • 使用 high-Vt 器件。
  • 使用 long-channel 器件。
  • well bias / body bias。
  • 使用 high-K 介质工艺.
  • power gating。

1.5 功耗中的主要占比

在 28/14/12/7/5nm 等 这类成熟工艺上，动态功耗仍然是主要部分，只不过静态功耗的占比在逐渐递增。

而动态功耗中，时钟树相关功耗会占到很大一部分，通常在15%-40%。所以几乎所有的soc都会做clock gating，即便不手动做，综合工具也会自动插入。

2. 全链路低功耗策略

2.1 设计层次与功耗收益

上层决定“要不要做、怎么做、做多少”；

下层则是在既定功能和架构下做局部优化。

例如：

  • 算法减少一次大规模访存，可能比后端优化几百个门更有价值。
  • 架构上减少跨模块搬运，可能比后面做 data gating 更彻底。
  • 系统调度让模块集中工作后快速休眠，可能比一直低频运行更省能量。
  • Memory 访问模式优化，可能比单纯换一个低功耗 SRAM compiler 选项更有效。

所以低功耗不是硬件一个阶段的任务，而应该从需求和架构评审阶段就开始着手。不过本篇文章主要围绕硬件低功耗手段进行展开。

2.2 系统架构层低功耗策略

系统层低功耗的核心问题是：哪些模块在什么场景下需要工作，哪些可以降频、降压、关时钟，甚至直接断电？

2.2.1 Power domain - Multi Voltage

电源域可以理解为具有同一种电源行为的一组集合。如可以将某个模块划为一个单独的电源域，或将若干子模块组成的一个子系统划为一个电源域。有了电源域，就可以执行相应的低功耗策略如电源关断和多电压设计。

Power gating 对静态功耗的收益很大，适合阶段性工作的大模块，引入 power gating 后，需要同步考虑：

  • 电源域划分是否合理。
  • 关断模块输出是否需要 isolation。
  • 关断前状态是否需要 retention。
  • 唤醒时序和状态恢复是否可控。
  • debug/test mode 下是否允许关断。
  • UPF/CPF、仿真、形式验证和后端实现是否完整支持。

如果某个模块只是短时间空闲，频繁 power on/off 反而可能得不偿失。Power gating 更适合“空闲时间足够长、状态恢复可控、关断收益明显”的大模块。对于一些小模块，仅关闭时钟即可。

动态功耗与电压平方相关，所以 multi-voltage 很有吸引力。比如非关键模块使用较低电压，时序紧张或性能敏感模块使用较高电压。但 multi-voltage 会引入新的工程问题：

  • 不同 voltage island 之间需要 level shifter。
  • 如果还叠加 power gating，可能需要 isolation + level shifter 的组合单元。
  • 跨电压域路径的时序约束更复杂。
  • 需要完整的 low power flow 支撑。
  • 后端实现和验证复杂度明显上升。

2.2.2 Clock domain - DFS

不同模块时钟要求不尽相同，合理分配时钟方案可降低非常客观的动态功耗。如为逻辑简单的模块分配较低频时钟，为高性能计算模块分配较高，或多档时钟。不同时钟域信号的交互需做好跨异步处理。

DFS是在时钟频率上进行动态调整，从而降低功耗的手段。如果动态调压风险太高，可以先做动态调频。动态频率调整不改变电压，收益比 DVFS 小，但工程风险也更低。

适合的做法是：

  • 给关键模块设计多个频率档位。
  • 按业务场景切换，比如高性能、普通、低负载模式。
  • 频率切换前等待当前事务完成。
  • 配合 thermal monitor 或 system monitor 做降频保护。

对于中等风险项目，DFS 往往比完整 DVFS 更容易落地。

2.2.3 DVFS/AVFS：电压频率综合优化手段

这里以DVFS为例进行说明。DVFS 的思想是让模块根据负载切换电压和频率。典型场景包括 CPU、NPU等有明显性能档位的模块。

例如视频场景中，不同分辨率和帧率对应不同算力需求：

  • 4K60 需要高频。
  • 4K30 可以降一档。
  • 1080P30 可能只需要更低频率。

DVFS 做得好，可以在“够用”的前提下减少多余性能。但实现上需要：

  • 性能监测或负载预测。
  • PMU/LDO/DCDC 配合。
  • 频率切换前后的保护时序。
  • 当前任务完成后的安全切换机制。
  • 软件调度策略。

DVFS 收益高，但验证、时序和系统调度成本也高，适合对功耗非常敏感且项目具备经验积累的场景。AVFS（Adaptive Voltage and Frequency Scaling）原理类似，通过自适应调整电压和频率来优化功耗。

2.2 处理器层低功耗策略

处理器层的低功耗不只是硬件问题，也和软件调度强相关。

常见手段包括：

  • sleep instruction。
  • shallow sleep：关时钟，保留电源，唤醒快，主要省动态功耗。
  • deep sleep：关时钟并关部分电源，省动态和静态功耗，但唤醒慢。
  • retention register 或 SRAM 保留关键状态。
  • 软件保存上下文，唤醒后恢复。
  • 异构架构，例如大核/小核、CPU/GPU/NPU 分工。
  • 提高 Cache hit rate，减少外部存储访问。

这里有一个工程判断：低功耗状态不是越深越好，而是要看 idle 时间能不能覆盖进入和退出低功耗状态的代价。如果只是短暂等待，关掉时钟可能就够了；如果长时间不用，才值得考虑 power gating。

2.3 模块层低功耗策略

即ic设计微架构范畴，其常见策略与soc系统层面基本一致，只不过规模更小。

  • 比较独立或大规模模块可划分为独立电源域/多级电源域进行管理，做关断 门控 多电压等。
  • 多级时钟门控策略，如模块层，子模块层，寄存器层等。
  • 微架构优化，数据通路不复位等。

这里对时钟门控进行展开：Clock gating 是数字芯片低功耗里最常用的手段。它的基本思想很简单，即模块或寄存器不工作时，不要让时钟继续翻转。

2.3.1 为什么 clock gating 很重要？

时钟网络会驱动大量寄存器和 clock tree buffer，即使数据不变化，时钟仍然在翻转。因此在很多设计中，时钟功耗会占动态功耗的显著比例。如果能把 idle 模块的时钟关掉，收益往往很直接。

2.3.2 自动插入 ICG：依赖 RTL 风格

综合工具可以根据 RTL 里的 enable 逻辑推断 clock gating，但前提是代码风格要友好。典型写法是寄存器只有在 valid 或 enable 为真时更新：

always @(posedge clk or negedge rst_n) begin
  if (!rst_n)
    q <= '0;
  else if (en)
    q <= d;
end

如果代码里每个分支都让寄存器更新，工具就很难插入 ICG。比如：

always @(posedge clk or negedge rst_n) begin
  if (!rst_n)
    q <= '0;
  else if (en)
    q <= d;
  else
    q <= other_value;
end

这类代码看起来功能没问题，但从低功耗角度看，寄存器在大多数周期仍然可能被迫翻转。

工程上常见检查项：

  • enable 是否稳定，是否会引入毛刺。
  • enable 逻辑是否太长，影响 gating 插入或时序。
  • 最小 bitwidth 是否合理，太小的寄存器强行 gating 可能收益不高。
  • ICG 插入位置是在寄存器末端，还是更靠近上游 clock tree。
  • clock gating 覆盖率是否达到目标，比如关键数据通路和大寄存器组覆盖充分。
  • 一般要求门控率在85%以上。

2.3.3 手动多级门控：适合模块级和模式级控制

综合工具推断的 ICG 往往靠近寄存器端，能减少寄存器翻转，但上游 clock tree 仍可能在动。

对于大模块、互斥工作模式、子模块层级明显的设计，可以在 RTL 或架构层手动插入更高层级的 clock gate。

例如：

  • 子模块 idle 时关子模块 clock。
  • 多个工作模式互斥时，只打开当前模式需要的模块。
  • Macro IP，如 memory、PLL 周边控制、硬化 IP，单独做 clock enable。

这类设计收益更大，但也要特别注意：

  • 门控控制信号不要是复杂组合逻辑，避免毛刺。
  • 需要确认 clock gating cell 的时序和约束。
  • 避免把数据信号直接当控制信号造成循环或不可控路径。
  • CDC/RDC 和 low power verification 要覆盖相关场景。

2.4. Memory 层低功耗策略

大多 SoC 项目里，SRAM 的面积和功耗占比非常高，尤其AI芯片。

因此 Memory 低功耗不能只靠 compiler 默认选项，而要从架构、生成、使用方式和访问模式一起优化。

2.4.1 使用角度：并非所有存储都要用 SRAM/ROM

小容量存储未必适合使用SRAM，需对比不同实现的面积、功耗，最终敲定。

  • SRAM/ROM实现。
  • 寄存器实现。
  • case/组合逻辑实现等。

2.4.2 生成角度：不要过度拆分

同样容量的 SRAM，拆分越多，外围电路、冗余边界、时钟、使能和后端布线成本可能越高。

因此要权衡：

  • 拆分后是否真的能降低访问功耗。
  • 是否增加了面积。
  • 是否增加 APR 难度。
  • 是否引入更多 enable、address、data 的翻转。
  • 是否让时序变复杂。

一般来说，拆分要服务于真实访问模式，而不是为了“看起来更灵活”。

2.4.3 功能角度：优先使用 Memory 自带低功耗模式

很多 SRAM macro 自带低功耗模式，例如：

  • sleep mode。
  • retention mode。
  • shutdown / power down mode。
  • quick nap。
  • selective precharge。
  • back bias。
  • multi-Vt 选项。

这些模式通常比外部硬凑逻辑更直接，但使用时要确认：

  • 低功耗模式下是否还能读写。
  • 内容是否保持。
  • 唤醒延迟是多少。
  • 控制信号时序要求是什么。
  • 是否允许降低供电电压。
  • 是否需要 isolation 或 retention。

对于大块 AO 区域 SRAM，内部是否支持关断和 retention，需要在 memory compiler 和工艺库层面提前确认。

2.4.4 访问模式角度：合并访问、提高 burst length

SRAM 的功耗不仅和容量有关，也和访问方式有关。

如果平均数据量不变，把频繁、零散的读写合并成更长 burst，通常可以减少平均动态功耗。因为每次访问都会带来外围电路、译码、字线、位线、enable 等活动，访问越碎，额外开销越明显。

常用优化包括：

  • grouping memory accesses。
  • avoiding redundant activity on enable pins。
  • 提高 burst length。
  • 减少无效读写。
  • 对 SRAM 端口做 clock gating。
  • 在允许的情况下，用 multi-cycle 降低 SRAM 访问频率。
  • 让数据位宽尽量匹配真实使用宽度，避免读出大量无用 bit。

Memory 低功耗最怕“功能已经做完，最后才发现访存模式很差”。这类问题后期很难救，最好在架构评审阶段就把数据流画清楚。

2.5. 物理实现层低功耗策略

到了综合、布局布线和工艺器件层，仍然有不少低功耗手段：

  • low power 工艺库。
  • high-Vt / low-Vt 混用。
  • long-channel 器件。
  • multi-bit cell。
  • clock tree 优化。
  • pin swap / cell merge / shorten。
  • well bias。
  • FD-SOI 或特殊低功耗工艺。

这些手段很重要，但它们更像“把既定架构实现得更好”。如果系统架构本身高频搬数据、Memory 访问碎片化、模块 idle 也一直跑时钟，后端优化很难彻底扭转局面。

此外还需考虑额外引入的工作量：

  • clock gating：动态功耗收益可观，实现难度低，时序和验证影响通常较小。
  • multi-Vt / multi-channel：主要降低静态功耗，不提升实现复杂度。
  • multi-voltage：动态和静态都有收益，但验证和实现复杂度上升。
  • power gating：静态功耗收益最大，验证、实现复杂度提升。
  • DVFS：收益很高，但系统和验证复杂度也最高。

没有免费的低功耗技术。每一种技术都在收益、面积、时序、验证和实现成本之间做权衡。

3. 基于 Golden UPF 的 EDA Flow

所谓golden upf，指的是整个流程下来用的都是同一套upf文件，维护起来比较简单。

此外还有一种upf flow，每个阶段不太一样，维护起来比较麻烦。

3.1 RTL 阶段

在系统架构确定之后，就可以开展UPF的设计工作。根据架构设计，在UPF中定义电源域、电源状态、电源约束、电源端口、电源连接、电源开关、电源隔离、电源保护等信息。

这个阶段还未进行综合，主要的保障手段通常为：

  • 通过rlt + upf 进行rtl阶段的clp/vclp检查。
  • 通过rtl + upf 进行vcs低功耗前仿。

3.2 综合阶段

综合阶段，综合工具读入 UPF 后，工具会根据 power domain、supply set、isolation、retention、level shifter、power switch 等约束插入或保留相关结构。如果相关语句错误，或部分macro的backup power没有显示连接，则工具会报相应错误。综合完后后会吐出网表文件，主要的保障手段通常为：

  • 通过 网表文件 + upf 进行网表阶段的clp/vclp检查。
  • 通过 网表文件 + upf 进行vcs低功耗仿真。
  • 通过 网表文件 + upf 进行一致性检查，如formality。

3.3 后端阶段

后端会输出包含完整电源信息的apr pr/pg网表，主要的保障手段通常为：

  • 通过 pg网表 + upf 进行PG阶段的clp/vclp检查。
  • 通过 pg网表 进行vcs低功耗后仿。
  • 通过 pr网表 + upf 进行一致性检查，如formality。

对于电源端口较多的复杂设计，formality比对会异常慢，可通过约束电源方向及固定值来极大加速比对过程。

3.4 常见 CLP/VCLP 问题

clp/vclp检查能同时暴露多个维度的问题，如upf，网表，甚至是lib库：

  • 综合或 APR 网表里出现散落单元，电源域归属不清晰。
  • UPF 中引用的 object 不存在，或层级路径和网表不一致。
  • 多轨电源 cell 的 backup supply 未显示连接。
  • 关断域输出跨到常开域时缺少 isolation，或 isolation clamp 值不合理。
  • 库中 pin 的 power domain 设置错误，导致工具判断 cell 供电关系异常。
  • 缺少 power state，或者 power state 定义和真实模式不匹配。
  • PSW 开关表达式交叠、不互补，或者多个 power state 下控制关系不唯一。

4 工程实现上的要点

4.1 典型组成

一个典型的低功耗实现方案包含以下要素：

  • 位于 always-on 域的PMU ，通过 APB/AHB 等总线暴露功耗管理寄存器。
  • 软件或固件配置功耗管理寄存器，产生 PSW_EN、ISO_EN、retention enable、clock gate enable 等控制信号。
  • 可关断模块内部的主电源由 power switch cell 控制，控制信号本身必须来自常开电源域。
  • 关断域输出到常开域或其他工作域时，需要 isolation cell 防止 X 态或未知电平向外传播。
  • 如果关断前需要保留状态，需要 retention register 或 retention SRAM，并明确 save / restore 时序。

这里最容易忽略的是：控制低功耗的信号链路本身不能随着被控模块一起断电。 例如 PSW 控制、isolation 控制、sleep/ack 握手、复位保持等信号，通常要由 AO 域供电，或者通过 AO buffer 维持。否则模块一旦关断，控制信号也失效，系统就无法可靠地再次唤醒它。

4.2 上下电顺序

Power gating 不是把 sleep 信号拉起来这么简单。关断和唤醒都要有明确顺序，否则容易出现 X 态传播、状态丢失、门控时钟异常、复位释放过早等问题。

一个常见的 Power Down 流程可以概括为：

  1. 停止向该模块发起新事务，等待正在进行的事务完成。
  2. 将模块相关软件复位或硬件复位置为有效，使内部状态进入可控状态。
  3. 关闭该模块相关门控时钟，避免关断过程中继续翻转。
  4. 使能 isolation，让关断域输出被钳位到安全值。
  5. 拉起 sleep / power down 请求，关闭 power switch。
  6. 等待 sleep_ack 或电源稳定状态反馈，确认关断完成。

对应的 Power Up 流程通常反过来，但不应简单机械反序：

  1. 取消 sleep / power down 请求，打开 power switch。
  2. 等待电源稳定或 sleep_ack 表示上电完成。
  3. 关闭 isolation，让输出重新由真实逻辑驱动。
  4. 打开模块门控时钟。
  5. 释放复位，允许模块重新参与系统事务。

实际项目里，复位、isolation、clock gate、sleep/ack 的具体先后要结合 IP 协议和低功耗 cell 要求定义，但原则很明确：先让模块安静下来，再隔离，再断电；先让电源稳定，再解除隔离、打开时钟、释放复位。

仿真和后仿里也要给上下电过程留足时间间隔。特别是后仿，power on/off、isolation enable、reset release、clock enable 如果挤在同一拍附近，很容易出现 X 传播、时序窗口不清晰或工具模型行为不一致的问题。

4.3 PSW 注意事项

Power switch cell 的作用是把全局电源连接到局部关断域电源。它的收益很直接，但实现上有两个常见坑：控制链路和上电浪涌。

首先，PSW 的 enable 链路通常需要 AO 供电。某些工艺库会提供 head buffer / AO buffer 这类 cell，用来把 HEADEN、SLEEP 等控制信号传递到关断域内部，同时保证即使局部 VDD 被关掉，控制信号路径仍然可控。

其次，大 power domain 需放缓其上电过程，否则会带来明显的 inrush current，可能造成 IR drop、电源噪声和唤醒失败。这里以双链psw为例：

  • 先打开小驱动 power switch管，再打开大驱动 power switch管。
  • 使用 daisy chain 或分组 enable，让 power switch 分批逐级开启。
  • 根据电源网络能力、domain 电容和唤醒时间要求，设置合理的上电间隔。
  • 后端和仿真需要能看到这些时序假设，否则版图和验证可能各按各的理解实现。

这也是为什么 power gating 不能只由 RTL 或 UPF 单点决定，它需要架构、设计、验证、后端和电源完整性分析一起闭环。

5. 结语：低功耗设计的本质是系统取舍

低功耗设计最难的地方，不是某个技术名词，而是取舍。

降低功耗可能带来面积增加；降低电压可能带来时序风险；power gating 能降漏电，但会引入状态保存和验证复杂度；DVFS 收益很高，但需要软硬件调度配合；Memory 拆分看似灵活，却可能增加面积、功耗和后端难度。

所以，一个成熟的 SoC 低功耗方案，应该回答清楚这些问题：

  • 哪些场景最耗电？
  • 哪些模块真正值得优化？
  • 是动态功耗为主，还是静态功耗为主？
  • 是要降平均功耗，还是要压峰值？
  • 哪些技术收益足够大，值得承担实现和验证成本？
  • 哪些优化只是“看起来很低功耗”，但实际收益有限？

低功耗不是把所有技术都堆上去，而是在正确的层次，用合适的复杂度，解决真实的功耗瓶颈。

如果用一句工程化的话收尾：先让系统少做事，再让模块少翻转，最后再让每一次翻转更便宜。
