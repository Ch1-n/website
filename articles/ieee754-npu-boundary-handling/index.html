NPU 设计中自研浮点单元的 IEEE 754 工程取舍

结合 IEEE 754-2019 标准语义，梳理 NPU 浮点单元中 zero、subnormal、inf、NaN、rounding、overflow、underflow、inexact 和 sticky flags 的硬件处理方式。

NPU设计IEEE 754NPU浮点单元2026-06-28

最近在整理 NPU 浮点计算单元里的 IEEE 754 边界处理逻辑，发现这一块细节还是比较多的，就索性梳理了下。Synopsys DesignWare 中当然有比较成熟的浮点运算 IP，例如 fp_add、fp_mul、fp_mac、fp_piped_add、fp_piped_mul 等。但对于自研 NPU 而言，直接使用通用浮点 IP 并不总是最优解。Tensor/Vector 路径通常会有自己的数据格式、吞吐目标、异常策略、旁路逻辑和功耗门控需求。如果把这些都交给通用 IP，面积、时序、功耗和调度灵活性未必合适。

因此，不少自研 NPU 会自己实现一套更贴合架构的数据格式转换和浮点运算单元。为了支持 IEEE 754，做浮点硬件时，就不得不考虑一些非正常值的处理：

  • 输入到底是 normal、subnormal、zero、inf 还是 NaN？
  • +0 和 -0 在计算路径中要不要完整保留？
  • 0 * inf、inf + (-inf)、sNaN 进入 MAC 后如何处理？
  • subnormal 是严格支持，还是做 DAZ / FTZ？
  • IEEE 754 的异常 flag 如何映射到 NPU 架构级状态寄存器？

这些问题看起来是“边界条件”，但在 NPU / AI 推理芯片里，它们会直接影响精度、 PPA、时序、吞吐率、软件可控性，以及模型数值稳定性。

本文会把内容分成两层：

  1. IEEE 754-2019 标准语义：标准原文明确规定了什么。
  2. NPU 工程取舍：为了实现高吞吐，硬件如何做简化。

需要先说明一点：IEEE 754-2019 定义了 binary16(即FP16)、binary32、binary64、binary128 等二进制浮点格式。BF16、TF32、FP8、FP4 这类 AI 低精度格式不是 IEEE 754-2019 的标准基本格式。工程上它们通常借鉴 IEEE 754 的 S/E/M 编码思想和特殊值分类方式，但具体 NaN、inf、subnormal、rounding 行为要以芯片 ISA、编译器和 runtime 约定为准。

1. IEEE 754-2019 相关章节

IEEE 754-2019 和硬件实现最相关的部分主要有几块：

协议章节关注点对硬件的直接影响
Clause 3.4浮点格式编码S/E/T 拆分，zero/subnormal/normal/inf/NaN 分类
Clause 3.6 Table 3.5binary16/32/64/128 参数位宽、precision、exponent width、bias
Clause 4.3rounding-direction attributesRNE、RTZ、RTP、RTN 等舍入模式
Clause 6.1infinity arithmeticinf 的传播和非法组合
Clause 6.2NaNqNaN、sNaN、payload、quieting
Clause 6.3sign bit+0/-0、积/商符号、精确零和舍入方向
Clause 7.1-7.6exceptions and flagsinvalid、divideByZero、overflow、underflow、inexact

这几部分基本决定了浮点单元的边界处理框架：

classify -> special-case decision -> main datapath -> normalize -> round -> pack -> flags

如果做的是 CPU FPU，那就需要尽量贴近完整协议语义。但 NPU 计算通路还不太一样，它更强调：

  • 高吞吐；
  • 固定 pipeline；
  • 多 lane 并行；
  • 低面积和低功耗；
  • 可接受一定程度的精度损失；
  • 异常可记录，但不能频繁打断流水。

因此下文在介绍的时候会写清楚：标准怎么说，以及 NPU 设计工程上怎么取舍。

2. IEEE 754 浮点数结构

IEEE 754-2019 Clause 3.4 定义了浮点数二进制的编码结构。一个二进制浮点数可以拆成：

S | E | T

其中：

  • S 是 1 bit 符号位；
  • E 是偏置后的指数位；
  • T 是尾数位；
  • precision p 包含隐含位，所以对 FP32 来说 p = 24，但显式 fraction 只有 23 bit。

以 binary32 为例：

S1E8T23
bias = 127
p = 24

Clause 3.4 中将所有浮点数分为以下五类：

条件类型有效数含义
E = 0, T = 0zero+0 或 -0
E = 0, T != 0subnormal无隐含前导 1
1 <= E <= 2^w - 2normal有隐含前导 1
E = 2^w - 1, T = 0infinity+inf 或 -inf
E = 2^w - 1, T != 0NaNqNaN 或 sNaN

其中qNaN 和 sNaN区别在于尾数最高位是否为1。

上述看起来可能不太清晰，这里以FP32进行举例说明：

条件类型有效数含义
E = 0, T = 0zero+0 或 -0
E = 0, T != 0subnormal无隐含前导 1
1 <= E <= 254normal有隐含前导 1
E = 255, T = 0infinity+inf 或 -inf
E = 255, T != 0NaNqNaN 或 sNaN

这意味着硬件 unpack 阶段不是简单把字段拆开，而是必须先做 class decode：

is_exp_zero = (E == 0);
is_exp_all1 = (&E);
is_frac_zero = (T == 0);

is_zero      = is_exp_zero && is_frac_zero;
is_subnormal = is_exp_zero && !is_frac_zero;
is_normal    = !is_exp_zero && !is_exp_all1;
is_inf       = is_exp_all1 && is_frac_zero;
is_nan       = is_exp_all1 && !is_frac_zero;

刚才也提到过，NaN 还要进一步区分 qNaN 和 sNaN。Clause 6.2.1 中给出了推荐编码方式：trailing significand field 的首位通常用于区分 quiet / signaling。

工程上可以写成：

is_qnan = is_nan && T[MSB];
is_snan = is_nan && !T[MSB];

这些分类信号会作为 sideband 跟随数据继续向后流动。对 NPU 来说，它们的意义很大：

  • 决定是否进入主乘法阵列；
  • 决定是否旁路输出常量；
  • 决定是否触发 invalid / overflow / underflow 等 sticky flags；
  • 决定是否做 clock gating 来减少无效翻转。

3. 常见格式与特殊值检测

IEEE 754-2019 Clause 3.6 Table 3.5 给出了浮点数的关键参数。

格式总位宽 kprecision pexponent bits wtrailing bits tbias
binary16161151015
binary323224823127
binary64645311521023
binary1281281131511216383

这里有一个容易混淆的地方：precision p 和显式尾数字段 t 不是一回事。

对于 normal number：

p = t + 1

因为 normal number 有一个隐含前导 1。

所以 binary32 虽然显式存储 23 bit 尾数，但 precision 是 24 bit。

AI 芯片里常见的格式更多，下表为标准IEEE 754格式之外的一些常用数据格式：

格式位宽典型编码bias
FP2424S1E8M15127
TF3219S1E8M10127
BF1616S1E8M7127
FP88S1E5M215
FP44S1E2M11

为什么说典型编码呢？因为协议没有明确指定，那么就完全可以按照需求定制格式，以FP8为例可以有S1E5M2和S1E4M3。

因此，在写文档的时候最好避免说“FP8/FP4 遵循 IEEE 754”。更准确的说法是：

FP8/FP4 采用类 IEEE 的 S/E/M 编码和特殊值分类思想，但具体特殊值、NaN、inf、subnormal 和 rounding 行为由架构自定义。

这句话非常重要。否则软件、验证和硬件会对“IEEE 754 支持程度”产生不同理解。

3.1 特殊值的检测

从硬件实现看，最适合做特殊值检测的位置通常不是 计算单元内部，而是 dequantize / unpack 输出侧。原因如下：

  1. 这里还保留原始格式信息，容易判断 E all 0、E all 1、M == 0、M != 0；
  2. 这里可以提前生成 sideband，让后级只处理统一语义。

一个比较实用的前端处理表如下：

输入分类前端动作数据动作状态动作
Zero生成内部 zero 常量保留符号，输出 +0/-0 或架构定义的单一 0一般不置异常
Inf生成内部 inf 常量输出同符号 inf；若目标格式不支持 inf，则按格式策略饱和或转 NaN一般不置异常
qNaN生成内部 qNaN 常量通常输出 canonical qNaN一般不置 invalid
sNaNquiet 成 qNaN将 NaN 尾数 quiet bit 置 1，再输出 qNaN置 invalid sticky flag
Subnormal按 CSR/格式策略处理严格模式保留；DAZ 模式强制为 signed zero可选置 underflow / inexact

很多NPU设计中，对sNaN的处理会采取“上报异常 且 静默降级”策略，可以把 sNaN 的处理集中在前端：

detect sNaN
    -> invalid pulse
    -> set quiet bit
    -> convert to qNaN
    -> downstream only sees qNaN

这样做的好处是，后级模块不再需要反复区分 sNaN/qNaN，只要传播 qNaN 和 sticky invalid 即可。否则 sNaN 如果一路穿过多级流水，可能在多个 stage 被重复识别，导致异常上报语义和验证模型都变复杂。

4. Zero

IEEE 754-2019 Clause 3.4 明确指出 signed zero：

E = 0, T = 0
value = (-1)^S * 0

Clause 6.3 又进一步规定了符号位在积、商、和、差以及精确零结果中的处理规则。

标准语义里，+0 和 -0 在数值比较上通常相等，但在某些运算中有区别：

1 / +0 = +inf
1 / -0 = -inf

精确零求和的符号也不是完全随意的。Clause 6.3 中规定，两个相反符号操作数相加得到精确 0 时，除 roundTowardNegative 外通常得到 +0；在 roundTowardNegative 下得到 -0。

这就是标准语义。

但在 NPU 设计中，根据需求可以酌情简化+0/-0。

例如一个 MAC tree 中，如果内部定点累加和精确抵消为 0，硬件可能直接根据 sum_is_zero 输出：

32'h00000000

也就是强制输出 +0。

这样可以省掉一部分符号归并和特殊舍入模式下的零符号判断。

但这属于工程降级，而不是 IEEE 754 的完整行为。因此需要在架构文档里说清楚：

  • 是否严格支持 signed zero；
  • 计算路径是否保留 -0；
  • 哪些 rounding mode 下允许强制 +0；
  • 软件是否能通过 CSR 切换 strict / fast 模式。

如果没有写清楚，验证会按 IEEE 754 reference model 报错，而硬件团队会觉得这是合理 PPA 取舍。这个坑很常见。

5. Inf

IEEE 754-2019 Clause 6.1 把 infinity 看作仿射意义上的无穷大：

-inf < finite numbers < +inf

很多含 infinity 的运算是可以正常传播的，例如：

finite + inf = inf
finite * inf = inf, finite != 0
finite / inf = 0
sqrt(+inf) = +inf

这些运算通常是 exact，不需要触发异常。

但 Clause 7.2 对 invalid operation 给出了明确条件，包括：

0 * inf -> invalid
inf * 0 -> invalid
fusedMultiplyAdd(0, inf, c) -> invalid, unless c is qNaN
inf - inf -> invalid
0 / 0 -> invalid
inf / inf -> invalid
sqrt(negative finite) -> invalid

默认异常处理下，invalid operation 的浮点结果应该是 qNaN。

假设硬件执行：

A * B + C

如果乘法路径遇到：

A = 0
B = inf

即使后面还有一个 + C，乘法部分已经触发 invalid。

如果乘积是 +inf，而 C = -inf，则加法路径遇到：

(+inf) + (-inf)

同样触发 invalid。

因此，非数处理不能只在最后 pack 阶段做一个结果检查。更稳妥的硬件结构是：

  1. unpack 阶段识别 zero、inf、NaN；
  2. 乘法前识别 zero * inf；
  3. FMA 中识别 inf + (-inf) 这类 inf 的加减；
  4. invalid 通过 sideband 向后传播；
  5. 最终 pack 阶段按优先级选择 qNaN / inf / zero / normal result。

5.1 工程问题示例：0 * inf

如果只看 IEEE 754 标准，0 * inf -> invalid -> qNaN 很清楚。

但 AI 模型里经常会故意构造 inf 或 -inf：

  • attention mask；
  • padding mask；
  • dropout mask；
  • softmax 前置屏蔽；
  • 某些稀疏或选择类算子里的哨兵值。

例如 attention 中常见做法是把 mask 位置加上 -inf，让 softmax 后概率变成 0。

如果某条融合路径中因为数据流重排、量化、矩阵计算或 mask 合并，出现：

0 * (-inf)

严格 IEEE 模式会生成 qNaN。这个 qNaN 一旦进入宽矩阵通路，可能会污染整行、整个 block，甚至影响后续 layer。

所以一些 NPU 会在 CSR 中提供动态策略位，

通过配置相关寄存器可实现：完全兼容协议模式 和快速模式。

对于完全兼容协议模式，硬件按 Clause 7.2 处理：

0 * inf -> invalid
result -> qNaN
invalid flag -> 1

适用于严格调试、数值验证、需要和 IEEE reference model 对齐的场景。

对于快速模式，硬件可以在特定算子路径中把 0 * inf 降级为 0 或 某一常量：

0 * inf -> 0

这适用于明确把 inf 当 mask 哨兵值使用的场景。

但这必须写进架构规范。因为它不是 IEEE 754 标准行为，而是 NPU 面向 AI 工作负载的工程覆盖策略。

我个人建议文档中把这种行为写成：

In strict IEEE mode, zero times infinity signals invalid and returns qNaN.
In tensor mask-tolerant mode, selected zero-infinity products may be squashed to signed zero under CSR control.

这样验证、编译器和 runtime 都有明确边界。

6. NaN

IEEE 754-2019 Clause 6.2 明确要求支持两类 NaN：

  • signaling NaN，也就是 sNaN；
  • quiet NaN，也就是 qNaN。

Clause 6.2.1 对 浮点数的 NaN 编码做了规定：

E = all 1
T != 0

此时该数是 NaN。推荐编码中，尾数的首位用于区分 qNaN 和 sNaN。工程上常用判断是：

is_nan  = exp_all_ones && frac_nonzero;
is_qnan = is_nan && frac_msb;
is_snan = is_nan && !frac_msb;

Clause 6.2 还规定：sNaN 作为操作数参与 general-computational 或 signaling-computational operation 时，会 signal invalid operation。默认异常处理下，如果需要给出浮点结果，结果应为 qNaN。对应到硬件，一般会这样做：

  1. unpack 阶段检测 sNaN；
  2. 拉高 invalid sideband；
  3. 将 sNaN quiet 成 qNaN；
  4. 后续流水只传播 qNaN；
  5. pack 阶段输出 canonical qNaN 或固定默认 qNaN。

IEEE 754 对 NaN payload 有诊断信息保留的建议。Clause 6.2.3 中提到，如果一个 NaN 输入被传播，结果最好保留其 payload；如果多个 NaN 输入同时出现，标准不指定选择哪个 payload。

但在 NPU 计算路径里，保留 payload 往往不划算。更常见的工程实现是直接输出固定默认 qNaN，例如 FP32 中常见的：

32'h7FC00000

这不是完整 payload-preserving 行为，但可以显著简化：

  • 多路 NaN 选择；
  • MAC array 内部旁路；
  • FMA 特殊值优先级；
  • pack 阶段 MUX；
  • 验证 reference model。

因此建议规格里写清：

Scalar FP path preserves NaN payload when possible.
Tensor path returns canonical default qNaN and only guarantees sticky invalid flag.

这样就把标准语义和 NPU 工程策略分开了。

7. Subnormal

IEEE 754-2019 Clause 3.4 中，E = 0, T != 0 表示 subnormal。它没有 normal 数的隐含前导 1。subnormal 的意义是实现 gradual underflow，让数值在接近 0 时不是突然断崖式清零，而是逐渐过渡。

标准语义很漂亮，但硬件代价并不低。严格支持 subnormal 通常意味着：

  • 乘法输入端需要识别 E = 0, T != 0；
  • 需要 LZD / LZC 做前导零检测；
  • 需要动态左移，让尾数规格化；
  • 指数需要补偿；
  • 加法输出端需要支持 subnormal 结果生成；
  • underflow 检测要和 rounding 结合；
  • pipeline 级数和关键路径都可能增加。

因此，高吞吐的NPU设计中通常会对该部分进行简化处理。

7.1 DAZ：Denormals Are Zero

输入端检测到 subnormal 后，直接把它当作 0：

subnormal input -> signed zero

这样可以省掉输入端规格化逻辑。

7.2 FTZ：Flush To Zero

如果输出端 rounded 值落入 subnormal 区间，直接输出 0：

subnormal output -> signed zero

这样可以省掉输出端渐进下溢处理。

但 DAZ / FTZ 不是 IEEE 754 严格行为。它们属于工程降级，需要明确说明：

  • 哪些格式启用 DAZ；
  • 哪些格式启用 FTZ；
  • 是否由 CSR 控制；
  • 是否影响 underflow / inexact flag；
  • 输入清零是否保留符号；
  • 输出清零是否保留符号。

对于 FP16 / BF16 / TF32，很多 NPU 会默认走 DAZ / FTZ 以换 PPA。

对于 FP8 / FP4，反而要更谨慎。因为低位宽格式本身表达范围小，如果再粗暴清零 subnormal，可能会显著影响量化误差。所以工程上可能出现：

  • FP16/BF16 Tensor 路径默认 DAZ / FTZ；
  • FP8/FP4 输入保留局部 subnormal 旁路；
  • compiler 禁止某些会丢 subnormal 的反量化路径；
  • FP8 反量化优先到 BF16，而不是 FP16，以保留动态范围。

这类策略如果不写清楚，软件以为数值还在，硬件入口已经清零，最后误差会很难定位。

8. Rounding

IEEE 754-2019 Clause 4.3 规定，除非另有说明，操作应当先得到无限精度、无限范围的中间结果，然后根据 rounding-direction attribute 舍入到目标格式。

标准定义了几类舍入方向：

  • roundTiesToEven：就近舍入，正好一半时舍入到最近的偶数，如2.5舍入到2，3.5舍入到4；
  • roundTiesToAway：就近舍入，正好一半时舍入到幅值更大的结果，即四舍五入；
  • roundTowardZero：向 0 舍入，也就是截断；
  • roundTowardPositive：向 +inf 舍入；
  • roundTowardNegative：向 -inf 舍入。

一般自己设计的FP单元通常不需要完整支持所有舍入模式。常见搭配是：

  • RNE：round to nearest, ties to even；
  • RTZ：round toward zero。

也就是上述的 roundTiesToEven 和 roundTowardZero。

9. 异常

IEEE 754-2019 Clause 7 定义了五类异常：

invalid operation
divideByZero
overflow
underflow
inexact

其中 Clause 7.1 说明了默认异常处理的基本思想：给出默认结果（常量）、继续执行，并设置对应 status flag。也就是说，标准本身并不要求每次浮点异常都精确打断程序。这点和 NPU 很契合。NPU 更适合用 sticky flags，而不是精确中断。

9.1 Overflow

Clause 7.4 定义 overflow：如果无界指数范围下的舍入结果超过目标格式最大有限值，就触发 overflow。默认结果取决于舍入方向：

  • RNE / roundTiesToAway：溢出到带符号 infinity；
  • RTZ：溢出到带符号最大有限值；
  • RTP / RTN：根据符号和方向决定 infinity 或最大有限值。

同时，默认 overflow 还会 signal inexact。

9.2 Underflow

Clause 7.5 定义 underflow：检测到 tiny non-zero result 时触发。对 binary format，tininess 可以选择 before rounding 或 after rounding，但一个实现中要保持一致。

标准还特别强调：underflow 的默认结果仍然是 rounded result，可能是 zero、subnormal，也可能是最小 normal。

更细的一点是：如果 underflow 后的 rounded result 是 exact，则默认处理下可以不 raise underflow flag。只有 inexact tiny result 才会把 underflow flag 和 inexact 一起拉起来。

9.3 Inexact

Clause 7.6 定义 inexact：如果 rounded 结果 和无限精度、无限范围下的数学结果不同，就 signal inexact。比如十进制里：

1 / 3 = 0.333333333...

如果只能保留 3 位小数：

1 / 3 = 0.333

这就是 inexact。这意味着只要舍入丢了信息，就可能有 inexact。

但在 NPU 里，如果每个 lane 都精确维护 inexact，成本会很高。工程上常见做法是：

  • 主路径内部生成简化版 inexact sideband；
  • lane 级 OR reduction；
  • block / kernel 结束后写 CSR sticky flag；
  • 软件按需读取，而不是每次异常都中断。

9.4 异常上报

IEEE 754-2019 Clause 7.1 要求每类异常都有对应 status flag，并且用户可以测试、修改、保存和恢复这些 flags。

在 NPU 里，我们通常不会做精确异常中断。原因很简单：

  • 计算单元数量太多；
  • 异常可能在任意 cycle 出现；
  • 精确中断会破坏固定流水；
  • AI kernel 更关心整体运行完成后的状态；
  • 许多异常只是诊断信息，不应该阻塞吞吐。

因此工程上通常做成 sticky flags：

lane_exception
    -> PE local OR
    -> tile/block OR
    -> engine-level sticky flags
    -> CSR readback

可以抽象成：

csr_invalid     <= csr_invalid     | block_invalid;
csr_div_by_zero <= csr_div_by_zero | block_div_by_zero;
csr_overflow    <= csr_overflow    | block_overflow;
csr_underflow   <= csr_underflow   | block_underflow;
csr_inexact     <= csr_inexact     | block_inexact;

这和 IEEE 754 的默认非停止异常处理精神并不冲突：默认情况下，异常被记录，计算继续执行。

真正需要定义清楚的是：

  • sticky flag 是按 lane、warp、tile、kernel 还是 engine 聚合；
  • flag 何时清零；
  • 多个 kernel 是否共享 flag；
  • 异常是否影响输出编码；
  • fast mode 下是否仍写 flag；
  • DAZ / FTZ 是否 raise underflow / inexact。

这些是 NPU 架构文档必须写清的内容。

9.5 异常的简化处理

如果输入已经是 NaN，或者某一级已经判定结果必然是默认 qNaN，那么继续让主乘法阵列、加法树、normalize、rounding 逻辑翻转，通常没有意义。一种常见的优化手段如下：

检测到特殊值输入，直接输出常量结果：

is_nan -> default_qnan
is_zero_product -> zero
is_inf_result -> signed_inf

对于 0 * inf，还要看当前 CSR 模式：

strict mode: qNaN + invalid
mask-tolerant mode: zero

10. 边界处理流程示例

把标准语义和工程取舍结合起来，一个 NPU 浮点单元可以这样组织。

Stage 0：Unpack / Classify

  • 拆出 S/E/T；
  • 根据 Clause 3.4 识别 zero、subnormal、normal、inf、NaN；
  • 对 NaN 区分 qNaN / sNaN；
  • 根据格式生成 bias、emin、emax；
  • 输出分类 sideband。

Stage 1：Policy Override

  • 根据 CSR 决定是否启用 DAZ；
  • 根据 CSR 决定是否启用 mask_en；
  • sNaN quiet 成 qNaN，并记录 invalid；
  • 对明确可旁路的输入生成 special-case 控制。

Stage 2：Main Datapath / Special Datapath

  • normal 数据进入 multiplier / adder / FMA；
  • zero * inf、inf - inf 等组合进入 special-case logic；
  • qNaN 直接传播或转成默认 qNaN；
  • invalid / overflow / underflow / inexact 以 sideband 形式向后流动。

Stage 3：Normalize

  • 对主路径结果做规格化；
  • 计算 E_calc；
  • 生成 GRS 位；
  • 检测 tiny / huge 候选状态；
  • 准备 rounding。

Stage 4：Round

  • RTZ：直接截断；
  • RNE：根据 G & (L | R | S) 判断是否进位；
  • 处理尾数进位导致的指数加一；
  • 检测 rounding 后 overflow；
  • 根据工程策略处理 subnormal output。

Stage 5：Pack / Commit

  • 按目标格式重新编码；
  • 根据优先级选择 normal、zero、inf、qNaN；
  • 聚合 lane-level flags；
  • 写入 CSR sticky flags；
  • 输出数据和状态。

其中最容易出 bug 的不是主乘法器，而是优先级。

例如一个 FMA 中同时存在 qNaN 输入、0 * inf、overflow 候选时，最后到底输出什么、是否写 invalid，要严格对齐标准和架构文档。Clause 7.2 对 FMA 的 0 * inf 还专门提到：如果加数 c 是 qNaN，invalid 是否 signal 是 implementation-defined。这样的细节必须在本架构中固定下来，否则验证会非常痛苦。

11. 文档说明

如果要把这套逻辑真正落到项目里，我建议至少要包含下面几张表。

1. Format table

说明每种格式的：

  • 位宽；
  • S/E/M；
  • bias；
  • 是否支持 inf；
  • 是否支持 NaN；
  • 是否支持 subnormal；
  • 默认 qNaN 编码；
  • 是否 IEEE 754 标准格式。

2. Special-case table

列出每种特殊组合：

0 * inf
inf + (-inf)
finite / 0
0 / 0
inf / inf
sNaN input
qNaN input
overflow
underflow
inexact

并明确：

  • 输出编码；
  • flag；
  • 是否受 CSR 覆盖；
  • strict mode / fast mode 是否不同。

3. Rounding table

说明每条路径支持哪些舍入：

  • scalar FP；
  • vector FP；
  • tensor / MMA；
  • dequantize；
  • requantize；
  • convert。

不要默认所有路径都支持 IEEE 754 的全部 rounding attributes。

4. DAZ / FTZ policy table

说明每种格式和每条路径：

  • input subnormal 是否 DAZ；
  • output subnormal 是否 FTZ；
  • 是否保留 sign；
  • 是否 raise underflow；
  • 是否 raise inexact；
  • 是否允许软件关闭。

5. Bypass priority table

说明每个算子或函数近似单元的特殊值旁路优先级：

  • qNaN 是否最高优先级；
  • sNaN 是否在入口统一 quiet；
  • inf 是传播、饱和还是转 NaN；
  • zero 是否能直接输出常量；
  • normal path 何时允许 clock gating。

这几张表比一句“支持 IEEE 754”更有价值。

12. 小结

IEEE 754-2019 给了我们一套非常清晰的浮点语义：

  • Clause 3 定义编码与格式；
  • Clause 4 定义舍入；
  • Clause 6 定义 infinity、NaN 和 sign bit；
  • Clause 7 定义异常和 status flags。

但 NPU 浮点单元不是简单复刻 CPU FPU。在高吞吐通路里，硬件要在几件事之间做取舍：

  • 标准一致性；
  • 精度；
  • AI 算子容错；
  • PPA；
  • 时序；
  • 吞吐率；
  • 软件可控性；
  • 验证复杂度。

我觉得工程上最重要的不是简单说“支持 IEEE 754”，而是把下面这些问题逐条定义清楚：

输入如何分类？
哪些格式是标准 IEEE 格式，哪些是工程格式？
NaN payload 是否保留？
sNaN 是否 quiet？
0 * inf 是否允许 CSR 覆盖？
subnormal 是否 DAZ / FTZ？
RNE / RTZ 支持在哪些路径？
overflow / underflow / inexact 如何写 sticky flag？
strict mode 和 fast mode 的输出是否不同？

这些边界定义清楚，浮点单元才真正从“符合标准的概念设计”，走到了“可以被 RTL、验证、编译器和软件共同使用的工程实现”。

感兴趣的可以去看一下IEEE 754-2019协议原文。
